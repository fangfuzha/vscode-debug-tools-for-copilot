# VS Code 调试工具 for Copilot MCP 工具说明

本文档详细说明本扩展在 `VS Code Debug Tools for Copilot` 服务下暴露的 MCP 工具。Copilot Chat 可以通过这些工具读取和控制当前工作区的 VS Code 断点，并完成常见的调试工作台操作。

English version: [doc/mcp-tools.en.md](mcp-tools.en.md)

## 概述

扩展通过 `vscode.lm.registerMcpServerDefinitionProvider` 注册本地 MCP 服务定义。

- 服务名称：`VS Code Debug Tools for Copilot`
- Provider ID：`fangfuzha.vscode-debug-tools-for-copilot`
- MCP 服务版本：`0.5.0`

## 提供的工具

### `list_breakpoints`

- 输入：`{}`
- 说明：列出当前工作区中所有已注册的 VS Code 断点，不做过滤、不做排序，返回当前内存中的完整断点集合。
- 输出：断点快照数组，每一项都可以用作后续修改或删除时的稳定引用。

### `list_breakpoints_in_file`

- 输入字段：
  - `filePath` (字符串，必填)：要列出断点的源文件路径，可以是绝对路径，也可以是相对于第一个工作区文件夹的相对路径。
- 说明：列出指定文件中的所有源断点。
- 输出：解析后的文件路径、工作区相对路径、数量以及匹配到的断点快照。

### `get_breakpoint_statistics`

- 输入：`{}`
- 说明：汇总当前断点集合，返回总数、启用数、禁用数以及按类型统计的数量。
- 输出：可用于确认批量操作结果或快速了解当前断点分布的统计对象。

### `add_source_breakpoint`

- 输入字段：
  - `filePath` (字符串，必填)：要下断点的源文件路径，可以是绝对路径，也可以是相对于第一个工作区文件夹的相对路径；空字符串不合法。
  - `line` (整数，必填)：断点所在行号，从 1 开始计数，不能为 0、负数或小数。
  - `column` (整数，可选)：断点所在列号，从 1 开始计数；不传时默认落在该行第 1 列。
  - `enabled` (布尔，可选)：新建断点是否立即启用；不传时默认 `true`。
  - `condition` (字符串，可选)：条件表达式，只有表达式结果为真时断点才会触发。
  - `hitCondition` (字符串，可选)：命中次数条件，用于控制第几次命中时才真正暂停。
  - `logMessage` (字符串，可选)：日志断点输出的消息内容；如果提供该字段，断点通常用于日志输出而不是暂停。
- 说明：在指定文件和位置添加一个源断点。
- 行为：通过 `vscode.debug.addBreakpoints` 添加断点，并返回新增断点的快照，便于后续继续使用同一个 `key`。

### `add_function_breakpoint`

- 输入字段：
  - `functionName` (字符串，必填)：要下断点的函数名，不能为空。
  - `enabled` (布尔，可选)：新建函数断点是否立即启用；不传时默认 `true`。
  - `condition` (字符串，可选)：条件表达式，只有表达式结果为真时断点才会触发。
  - `hitCondition` (字符串，可选)：命中次数条件，用于控制第几次命中时才真正暂停。
  - `logMessage` (字符串，可选)：日志断点输出的消息内容。
- 说明：在指定函数上添加一个函数断点。
- 行为：通过 `vscode.debug.addBreakpoints` 添加断点，并返回新增断点的快照，便于后续继续使用同一个 `key`。

### `remove_breakpoint`

- 输入字段：
  - `key` (字符串，可选)：断点的稳定标识，优先使用；如果提供了它，系统会按完整断点描述精确匹配。
  - `filePath` (字符串，可选)：目标断点所在文件路径，当没有 `key` 时必须提供。
  - `line` (整数，可选)：目标断点所在行号，从 1 开始；当没有 `key` 时必须提供。
  - `column` (整数，可选)：目标断点所在列号，从 1 开始；只在同一文件同一行存在多个断点时才建议传入。
  - `functionName` (字符串，可选)：目标函数名，当没有 `key` 时也可以通过函数名删除函数断点。
- 说明：删除匹配的断点。该工具可以通过 `key` 精确删除，也可以通过文件路径 + 行号定位删除。
- 规则：如果提供了 `key`，优先按 `key` 查找；否则如果提供了 `functionName`，会按函数名查找函数断点；否则必须同时提供 `filePath` 和 `line`。`column` 仅用于进一步缩小匹配范围，不提供时会匹配该行的所有断点。

### `set_breakpoint_enabled`

- 输入字段：
  - `key` (字符串，可选)：断点的稳定标识，优先使用；如果提供了它，系统会按完整断点描述精确匹配。
  - `filePath` (字符串，可选)：目标断点所在文件路径，当没有 `key` 时必须提供。
  - `line` (整数，可选)：目标断点所在行号，从 1 开始；当没有 `key` 时必须提供。
  - `column` (整数，可选)：目标断点所在列号，从 1 开始；用于在同一行存在多个断点时精确定位。
  - `functionName` (字符串，可选)：目标函数名，当没有 `key` 时也可以通过函数名修改函数断点状态。
  - `enabled` (布尔，必填)：目标状态，`true` 表示启用，`false` 表示禁用。
- 说明：启用或禁用匹配的断点。
- 行为：先移除旧断点，再以目标启用状态重新添加。这样可以保持断点位置、条件、命中条件和日志消息不变，只修改启用状态。

### `update_breakpoint`

- 输入字段：
  - `key` (字符串，可选)：断点的稳定标识，优先使用；如果提供了它，系统会按完整断点描述精确匹配。
  - `filePath` (字符串，可选)：目标断点所在文件路径，当没有 `key` 时必须提供。
  - `line` (整数，可选)：目标断点所在行号，从 1 开始；当没有 `key` 时必须提供。
  - `column` (整数，可选)：目标断点所在列号，从 1 开始；用于在同一行存在多个断点时精确定位。
  - `functionName` (字符串，可选)：目标函数名，当没有 `key` 时也可以通过函数名定位函数断点。
  - `enabled` (布尔，可选)：新的启用状态。
  - `condition` (字符串 | null，可选)：新的条件表达式；传入 `null` 可清空已有条件。
  - `hitCondition` (字符串 | null，可选)：新的命中次数条件；传入 `null` 可清空已有命中条件。
  - `logMessage` (字符串 | null，可选)：新的日志消息；传入 `null` 可清空已有日志消息。
- 说明：在不改变断点位置或函数名的前提下更新匹配断点的属性。
- 行为：工具会先移除匹配断点，再按新的属性重新添加。未显式提供的字段会被保留，传入 `null` 则会清空对应的可选文本字段。如果某个匹配断点无法重新创建，会保留原样并在结果中的 `unsupported` 字段里返回。

### `add_function_breakpoint` 与函数断点

- `functionName` 只对 `function` 类型断点有意义。
- 如果你已经拿到了 `key`，优先使用 `key`，这样可以准确区分同名但条件不同的断点。
- 如果只有函数名，没有 `key`，`remove_breakpoint` 和 `set_breakpoint_enabled` 都会匹配同名函数断点。

### `clear_breakpoints`

- 输入：`{}`
- 说明：移除当前工作区中所有已注册的 VS Code 断点，不区分类型、不区分文件、不保留任何断点。

### `set_breakpoints_enabled_in_file`

- 输入字段：
  - `filePath` (字符串，必填)：要批量处理的源文件路径，可以是绝对路径，也可以是相对于第一个工作区文件夹的相对路径。
  - `enabled` (布尔，必填)：目标状态，`true` 表示启用该文件中的所有源断点，`false` 表示全部禁用。
- 说明：启用或禁用指定文件中的所有源断点。这个工具只影响 `source` 类型断点，不会修改函数断点或其他未知类型断点。

### `search_breakpoints`

- 输入字段：
  - `query` (字符串，必填)：要搜索的关键字，会在断点的类型、状态、文件路径、相对路径、条件、命中条件、日志消息、函数名和标签等字段中进行模糊匹配。
  - `kind` (`source` | `function` | `unknown`, 可选)：按断点类型过滤，只返回指定类型的断点。
  - `enabled` (布尔，可选)：按启用状态过滤，`true` 只看启用中的断点，`false` 只看已禁用的断点。
  - `limit` (整数，可选，最大 100)：最多返回多少条结果；不传时返回全部匹配项。
- 说明：根据文本、断点类型和启用状态搜索断点。这个工具适合在你只知道断点大致特征、但不确定 `key` 时使用。
- 输出：返回匹配断点快照以及数量和查询元数据，便于继续选择、删除或切换状态。

### `list_debug_sessions`

- 输入：`{}`
- 说明：列出扩展已跟踪到的调试会话，并返回每个会话的 `sessionId`。
- 输出：会话快照数组，包含会话 ID、名称、类型、工作区路径、是否为活动会话，以及最近一次已知的停止线程 ID。
- 说明：这个结果是尽力收集的，不保证覆盖扩展激活前就已经存在且未激活的会话。

### `list_debug_threads`

- 输入字段：
  - `sessionId` (字符串，可选)：目标调试会话的 ID；不传时默认操作当前活动调试会话。
- 说明：列出指定调试会话已知的线程，并返回每个线程的 `threadId`。
- 输出：线程快照数组，包含线程 ID、线程名称，以及所属会话的基础信息。

### `pause_debug_session`

- 输入字段：
  - `sessionId` (字符串，可选)：目标调试会话的 ID；不传时默认操作当前活动调试会话。
- 说明：暂停当前活动调试会话，使程序进入断点调试状态。
- 行为：等价于 VS Code 调试工具栏中的暂停按钮，只作用于当前活动调试会话。

### `continue_debug_session`

- 输入字段：
  - `sessionId` (字符串，可选)：目标调试会话的 ID；不传时默认操作当前活动调试会话。
- 说明：继续运行当前活动调试会话，直到下一个断点或程序结束。
- 行为：等价于 VS Code 调试工具栏中的继续按钮，只作用于当前活动调试会话。

### `step_over_debug_session`

- 输入字段：
  - `sessionId` (字符串，可选)：目标调试会话的 ID；不传时默认操作当前活动调试会话。
- 说明：单步跳过当前代码行，不进入函数内部，直接执行到下一行。
- 行为：等价于 VS Code 调试工具栏中的 Step Over，只作用于当前活动调试会话。

### `step_into_debug_session`

- 输入字段：
  - `sessionId` (字符串，可选)：目标调试会话的 ID；不传时默认操作当前活动调试会话。
- 说明：单步步入当前代码行调用的函数，从函数内部开始调试。
- 行为：等价于 VS Code 调试工具栏中的 Step Into，只作用于当前活动调试会话。

### `step_out_debug_session`

- 输入字段：
  - `sessionId` (字符串，可选)：目标调试会话的 ID；不传时默认操作当前活动调试会话。
- 说明：从当前函数中单步跳出，回到调用处的下一行。
- 行为：等价于 VS Code 调试工具栏中的 Step Out，只作用于当前活动调试会话。

### `restart_debug_session`

- 输入字段：
  - `sessionId` (字符串，可选)：目标调试会话的 ID；不传时默认操作当前活动调试会话。
- 说明：重启选定的调试会话，从头重新运行程序。
- 行为：工具会先停止选定会话，再使用该会话的已解析配置重新拉起，而不是依赖调试适配器自定义的 restart 请求。如果重新拉起失败，工具会返回错误。

### `stop_debug_session`

- 输入字段：
  - `sessionId` (字符串，可选)：目标调试会话的 ID；不传时默认操作当前活动调试会话。
- 说明：停止当前活动调试会话，结束调试。
- 行为：等价于 VS Code 调试工具栏中的 Stop，只作用于当前活动调试会话。

### `open_debug_source_file`

- 输入字段：
  - `filePath` (字符串，必填)：要切换到的源文件路径，可以是绝对路径，也可以是相对于第一个工作区文件夹的相对路径。
  - `line` (整数，可选)：要聚焦的行号，从 1 开始。
  - `column` (整数，可选)：要聚焦的列号，从 1 开始。
- 说明：在活动编辑器中打开并聚焦目标源文件，方便把调试上下文切到对应代码位置。
- 输出：打开后的文件路径、工作区相对路径、行列号和打开结果。

### `list_debug_launch_configurations`

- 输入字段：
  - `workspaceFolderPath` (字符串，可选)：只列出指定工作区文件夹中的 `.vscode/launch.json` 配置；不传时会遍历当前工作区里的所有文件夹。
- 说明：列出 `launch.json` 中的静态调试配置和 compound 配置，并附带扩展在运行期间观察到的最近调试配置缓存。它不会主动枚举所有动态 `DebugConfigurationProvider` 的即时结果。
- 输出：配置快照数组，包含名称、类型、来源文件、工作区路径以及原始配置对象；其中 `recentSession` 条目只是观察缓存，不代表 provider 的完整清单。

### `start_debug_launch_configuration`

- 输入字段：
  - `workspaceFolderPath` (字符串，可选)：当按名称启动时，用于消除多工作区下的同名配置歧义。
  - `launchName` (字符串，可选)：要启动的调试配置或 compound 名称。
  - `launchConfiguration` (对象，可选)：从 `list_debug_launch_configurations` 返回的原始配置对象；传入后会直接用它启动。
  - `noDebug` (布尔，可选)：是否以不调试模式启动。
- 说明：根据 `launch.json` 中的配置或者原始配置对象直接启动调试。
- 输出：启动结果、已启动会话快照以及所用工作区信息。

### `list_debug_call_stack`

- 输入字段：
  - `sessionId` (字符串，可选)：目标调试会话的 ID；不传时默认操作当前活动调试会话。
  - `threadId` (整数，可选)：目标线程 ID；不传时会优先使用当前选中的线程或自动解析一个可用线程。
  - `startFrame` (整数，可选)：从第几帧开始取，默认 `0`。
  - `levels` (整数，可选)：最多返回多少帧，默认 `20`。
- 说明：列出指定线程的调用堆栈，适合在暂停后定位当前执行点。
- 输出：线程 ID、会话快照、堆栈帧数量以及帧快照数组。

### `list_debug_scopes`

- 输入字段：
  - `sessionId` (字符串，可选)：目标调试会话的 ID；不传时默认操作当前活动调试会话。
  - `threadId` (整数，可选)：线程 ID；不传时会尝试从当前选中的栈帧或调用堆栈自动解析。
  - `frameId` (整数，可选)：栈帧 ID；不传时会尝试从当前选中的栈帧或调用堆栈自动解析。
- 说明：列出某个栈帧下的作用域，例如局部变量、参数和全局作用域。
- 输出：线程 ID、帧 ID、作用域数量以及作用域快照数组。

### `list_debug_variables`

- 输入字段：
  - `sessionId` (字符串，可选)：目标调试会话的 ID；不传时默认操作当前活动调试会话。
  - `variablesReference` (整数，必填)：要展开的变量引用值，通常来自 `list_debug_scopes` 或上一级变量结果。
- 说明：展开某个变量节点下的子变量。
- 输出：变量引用、变量数量以及变量快照数组。

### `evaluate_debug_expression`

- 输入字段：
  - `sessionId` (字符串，可选)：目标调试会话的 ID；不传时默认操作当前活动调试会话。
  - `expression` (字符串，必填)：要执行的表达式。
  - `threadId` (整数，可选)：目标线程 ID。
  - `frameId` (整数，可选)：目标栈帧 ID。
  - `context` (字符串，可选)：表达式上下文，可取 `watch`、`repl`、`hover` 或 `clipboard`。
- 说明：在选定调试上下文中执行表达式，适合快速求值和临时检查变量。
- 输出：求值结果、类型、变量引用以及调用上下文信息。

### `list_debug_watch_expressions`

- 输入：`{}`
- 说明：列出扩展自己维护的 watch 表达式列表。
- 输出：watch 表达式数组和数量。

### `add_debug_watch_expression`

- 输入字段：
  - `expression` (字符串，必填)：要加入 watch 列表的表达式。
- 说明：把一个表达式加入扩展维护的 watch 列表，后续可以统一刷新。
- 输出：新增的 watch 表达式快照以及当前 watch 列表。

### `remove_debug_watch_expression`

- 输入字段：
  - `watchExpressionId` (字符串，必填)：要移除的 watch 表达式 ID。
- 说明：从扩展维护的 watch 列表中移除指定表达式。
- 输出：是否移除成功以及移除后的 watch 列表。

### `clear_debug_watch_expressions`

- 输入：`{}`
- 说明：清空扩展维护的全部 watch 表达式。
- 输出：被清空的表达式数量和清空后的 watch 列表。

### `evaluate_debug_watch_expressions`

- 输入字段：
  - `sessionId` (字符串，可选)：目标调试会话的 ID；不传时默认操作当前活动调试会话。
  - `threadId` (整数，可选)：目标线程 ID。
  - `frameId` (整数，可选)：目标栈帧 ID。
- 说明：在选定调试上下文里批量求值扩展维护的 watch 表达式。
- 输出：每个 watch 表达式的求值结果、错误信息和上下文信息。

## 断点快照格式

每个返回的断点快照可能包含：

- `key`：稳定标识，用于在后续工具调用中精确定位同一个断点。
- `kind`：断点类型，只会是 `source`、`function` 或 `unknown`。
- `enabled`：当前断点是否启用。
- `condition`：条件表达式，表示断点只在条件成立时触发。
- `hitCondition`：命中次数条件，表示断点需要命中多少次后才真正生效。
- `logMessage`：日志断点输出的文本内容。
- `filePath`：源断点所在文件的绝对路径，仅源断点有值。
- `workspaceRelativePath`：源断点相对于当前工作区的路径，仅源断点有值，适合在 UI 里展示给用户。
- `line`：源断点所在行号，从 1 开始。
- `column`：源断点所在列号，从 1 开始。
- `functionName`：函数断点对应的函数名，仅函数断点有值。
- `label`：未知断点的标签或构造器名称，仅未知类型断点有值。

## 调试会话控制说明

- 这些调试会话工具都支持可选的 `sessionId`；不传时默认作用于当前活动调试会话。
- 如果 `sessionId` 无效，工具会返回错误，而不是尝试猜测目标会话。
- 列出线程时，优先先选定会话，再调用 `list_debug_threads` 获取线程 ID。
- 扩展在 `onDebug` 时激活，以尽量提前开始跟踪会话；但历史会话仍可能无法完整回收。
- 它们对应的是 VS Code 调试工具栏中的常见控制按钮，而不是断点增删改工具。
- `list_debug_launch_configurations` 只读取工作区里的 `.vscode/launch.json`；如果调试配置来自其他动态 provider，当前不会自动枚举。
- `list_debug_call_stack`、`list_debug_scopes`、`list_debug_variables` 和 `evaluate_debug_expression` 更适合在暂停状态下使用。
- watch 表达式由扩展自己维护，不是直接读取 VS Code 侧边栏里的原生 Watch 面板。
- `list_debug_launch_configurations` 只保证读取 `.vscode/launch.json` 和扩展运行期间观察到的最近配置；如果某个动态 provider 的配置从未被扩展观察到，就不会出现在结果里。
- 不要把 `list_debug_launch_configurations` 理解成“完整枚举所有动态 provider 当前可用配置”，它只是一个可用的 best-effort 视图。

## 稳定 `key` 生成规则

`key` 的生成方式如下：

1. 将断点描述对象序列化为 JSON
2. 对 JSON 字符串进行 Base64URL 编码

描述对象会尽量包含影响断点身份的字段，例如 `kind`、`filePath`、`line`、`column`、`functionName`、`condition`、`hitCondition`、`logMessage` 和 `label`。这意味着同一位置、同一条件的断点会生成相同的稳定 `key`，适合在后续修改和删除时复用。

## 文件路径处理

- `filePath` 可以是绝对路径，也可以是相对于第一个工作区文件夹的相对路径。
- 如果未打开工作区文件夹，则相对路径会被拒绝并抛出错误，因为无法判断相对路径的基准目录。
- 比较时会对路径进行标准化，在 Windows 上还会统一小写，以避免大小写差异导致的误判。
- 建议在自动化调用时优先传绝对路径，只有在明确知道工作区结构时再传相对路径。

## 说明

- MCP 工具直接操作 VS Code 调试器的断点集合，因此对断点的增删改会立即反映到 VS Code 的调试视图中。
- 相对路径按第一个工作区文件夹解析，多个工作区时不会自动遍历所有工作区文件夹。
- 如果没有匹配到断点，工具会返回空结果集，不会报错；如果参数本身不合法，则会抛出错误。
